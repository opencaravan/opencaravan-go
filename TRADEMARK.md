# Trademark Policy

"OpenCaravan" is a trademark of the OpenCaravan project. The Apache License
2.0 covers the source code in this repository; this policy covers the protocol
name and brand.

OpenCaravan is meant to be implemented widely. The point of this policy is to
keep the name a reliable signal of *the protocol*, while making it easy for
implementations to advertise compatibility.

## You may, without asking

- Describe your software as **"an OpenCaravan implementation"**, **"compatible
  with OpenCaravan"**, **"speaks the OpenCaravan protocol"**, or similar
  factual statements.
- Use the name "OpenCaravan" in documentation, talks, blog posts, conformance
  reports, comparison tables, and academic work.
- Import this Go module under its canonical path,
  `github.com/opencaravan/opencaravan-go`, and depend on it from your project.
- Redistribute unmodified source under the Apache License 2.0, keeping the
  project name as-is.
- Translate this package to other languages and call your port an
  "OpenCaravan implementation" — but please give the port a distinct name
  (for example, `opencaravan-rs`, `opencaravan-swift`) rather than reusing
  this module's name.

## You may not, without prior written permission

- Use "OpenCaravan" or a confusingly similar name as the name of a forked
  protocol, a product, a company, or a hosted service.
- Use the OpenCaravan name in a way that implies endorsement, certification,
  or official status that does not exist.
- Claim that an implementation is OpenCaravan-compliant or
  OpenCaravan-certified when it is not.
- Distribute a modified protocol vocabulary or an incompatible wire format
  under the "OpenCaravan" name. Forks of the protocol need a new name.

If you maintain a derivative protocol, please give it a distinctive name. You
are welcome to describe it as "derived from OpenCaravan" or "inspired by
OpenCaravan."

## Questions or permission requests

Open an issue at <https://github.com/opencaravan/opencaravan-go/issues> or
contact the project maintainers. Reasonable requests are usually granted.

This policy may evolve as the protocol matures. Material changes will be noted
in the project changelog.
